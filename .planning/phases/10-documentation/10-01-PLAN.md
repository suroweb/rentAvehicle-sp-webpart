Create `docs/` directory and write `docs/app-registration.md` -- a step-by-step Entra ID app registration guide.

**Audience:** SharePoint Developers, Azure Architects. Professional, direct tone like Azure/Microsoft official docs.

**Structure:**
1. **Title and introduction** -- one paragraph explaining what this guide sets up and why
2. **Prerequisites** -- M365 tenant with admin access, SharePoint admin role, Entra ID admin role. List but do not explain how to obtain them.
3. **Step 1: Create App Registration** -- exact menu paths in Entra admin center (Entra ID > App registrations > New registration). Name: `RentAVehicle-API`. Supported account types: single tenant. Redirect URI: leave blank initially.
4. **Step 2: Configure API Permissions** -- add these 4 Application permissions (NOT delegated):
   - `Calendars.ReadWrite` -- resource + personal calendar events
   - `Mail.Send` -- booking confirmation emails
   - `User.Read.All` -- location sync from Entra ID, manager lookup
   - `TeamsActivity.Send` -- Teams activity feed notifications (also needs RSC in manifest)
   - Include a > [!NOTE] callout: Application permissions require admin consent. Click "Grant admin consent for [tenant]".
5. **Step 3: Expose an API** -- this is the CRITICAL step most guides miss:
   - Set Application ID URI to `api://<your-client-id>`
   - Add scope: `user_impersonation` (Admin consent required, display name: "Access RentAVehicle API")
   - Pre-authorize SharePoint client: `00000003-0000-0ff1-ce00-000000000000` for `user_impersonation` scope
   - Include a > [!WARNING] callout about SPFx AadHttpClient failing silently if this step is skipped
6. **Step 4: Create Client Secret** -- Certificates & secrets > New client secret. Note expiry. Extract from `graphService.ts` that code uses `ClientSecretCredential` with `AZURE_CLIENT_ID`, `AZURE_TENANT_ID`, `AZURE_CLIENT_SECRET`.
7. **Step 5: Configure App Roles** -- extract role names from `auth.ts` ROLE_HIERARCHY: Admin, Manager, Employee. Document how to create each in "App roles" blade and assign users.
8. **Step 6: Configure Azure Functions Easy Auth** -- reference `auth.ts` x-ms-client-principal header parsing. Document the Azure portal path: Function App > Authentication > Add identity provider > Microsoft Entra ID. Set Client ID to match the app registration.
9. **Step 7: Environment Variables** -- extract the FULL list from `local.settings.template.json`. Present as a table with Name | Description | Where to Find columns. Use exact env var names. Mark tenant-specific values with `<your-value>` placeholders.
10. **Verification** -- after completing all steps, what to check: admin consent granted (green checkmarks), Application ID URI set, secret created (note value before leaving page).

**Format rules (from CONTEXT.md locked decisions):**
- Step-by-step numbered instructions with exact menu paths and values
- Text-only (no screenshots)
- Use GitHub callout blocks (> [!NOTE], > [!WARNING], > [!TIP]) for important information
- Include exact API permission scopes, redirect URIs, and configuration values from the actual codebase
- Use "Microsoft Entra ID" not "Azure AD" in all text (except when referencing legacy error messages)

Write `docs/deployment.md` -- a deployment guide covering SPFx package deployment, Azure Functions setup, database setup, and Teams tab configuration.

**Audience:** SharePoint Developers, Azure Architects. Professional, direct tone.

**Structure:**
1. **Title and introduction** -- one paragraph explaining what this guide covers (end-to-end deployment)
2. **Prerequisites** -- completed app registration (link to `app-registration.md`), SharePoint App Catalog admin access, Azure subscription, Node.js 22
3. **Part 1: Build the SPFx Package**
   - Clone repo, `cd spfx`, `npm install`
   - Extract exact build commands from `spfx/package.json` scripts section
   - Output: `sharepoint/solution/renta-vehicle.sppkg` (extract path from `package-solution.json` paths.zippedPackage)
   - > [!TIP] about production build flags if applicable
4. **Part 2: Deploy to SharePoint App Catalog**
   - Navigate to SharePoint Admin Center > More features > Apps > App Catalog
   - Upload `renta-vehicle.sppkg`
   - Trust the package (check "Make this solution available to all sites in the organization" -- extract `skipFeatureDeployment` from package-solution.json)
   - > [!NOTE] about webApiPermissionRequests: the package requests `user_impersonation` scope on `RentAVehicle-API`. Admin must approve in SharePoint Admin Center > API access.
5. **Part 3: Approve API Permissions**
   - SharePoint Admin Center > Advanced > API access
   - Approve pending request for `RentAVehicle-API` / `user_impersonation`
6. **Part 4: Deploy Azure Functions**
   - `cd api`, `npm install`, `npm run build`
   - Extract build/start commands from `api/package.json`
   - Configure environment variables per the app registration guide
   - CORS origins: extract from `local.settings.template.json` Host.CORS section -- include tenant SharePoint domain
   - Database setup: run `node setup-db.js` -- document what it creates
7. **Part 5: Vehicle Resource Mailboxes**
   - Reference `scripts/provision-vehicle-mailbox.ps1`
   - Document Exchange Online PowerShell commands for creating equipment mailboxes
   - > [!TIP] about the provisioning script automating this
8. **Part 6: Add to SharePoint Site**
   - Navigate to target site > Add page or edit page > Add web part > Search "RentAVehicle"
   - Or: Site contents > New > App > RentAVehicle
9. **Part 7: Deploy as Teams Tab**
   - > [!WARNING] Do NOT use "Sync to Teams" from the App Catalog if you need custom activity types. The auto-generated manifest strips `webApplicationInfo` and `activities` sections.
   - Document the TeamsSPFxApp.zip approach from Phase 9 session notes:
     a. The custom `teams/manifest.json` in the SPFx project defines activity types and webApplicationInfo
     b. `webApplicationInfo.id` MUST match the backend Entra app registration client ID (not SharePoint Client Extensibility Principal)
     c. Package as TeamsSPFxApp.zip or deploy via Teams Admin Center
   - Teams manifest app ID: extract from `spfx/teams/manifest.json`
   - Document version increment requirement (both SPFx package version and Teams manifest version must increment for update detection)
10. **Troubleshooting** section with Problem > Cause > Fix pattern:
    - "An OAuth permission with the scope user_impersonation could not be found" -- missing Expose an API config
    - "403 Forbidden on Graph API calls" -- missing admin consent or wrong permission type (delegated vs application)
    - "Teams notifications return 403" -- auto-synced manifest stripped webApplicationInfo; redeploy with custom manifest
    - "SPFx webpart not appearing in web part gallery" -- package not trusted or site-level feature not activated
    - "AadHttpClient returns 401" -- client ID mismatch between SPFx webApiPermissionRequests and Entra app registration
    - Any other gotchas from Phase 9 session notes

**Format rules (from CONTEXT.md locked decisions):**
