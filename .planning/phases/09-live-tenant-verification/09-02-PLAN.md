  <what-built>
The calendar service (calendarService.ts) is already implemented. It creates events via Graph API `/users/{email}/events` endpoint for both the vehicle resource mailbox and the employee's personal calendar. Events are created fire-and-forget when a booking is created via the booking endpoint.

This task verifies the create-booking calendar sync works against the live tenant.
  </what-built>
  <how-to-verify>
**Prerequisites:** API must be running (`cd api && npm start`). Test vehicle must have a resource mailbox email linked. Test user must have LOCAL_DEV_EMAIL set to their real email.

**Step 1: Create a booking via the API**
Using curl or the app UI, create a booking for the test vehicle:
```bash
curl -X POST http://localhost:7071/api/bookings \
  -H "Content-Type: application/json" \
  -d '{
    "vehicleId": <VEHICLE_ID>,
    "startTime": "<TOMORROW_9AM_UTC_ISO>",
    "endTime": "<TOMORROW_5PM_UTC_ISO>"
  }'
```
(Replace VEHICLE_ID with the test vehicle's ID, and dates with valid future dates in ISO 8601 format.)

Note the booking ID from the response.

**Step 2: Check the Azure Functions console output**
Look for log messages:
- "Calendar sync" related messages (success or error)
- Any Graph API error messages

**Step 3: Verify resource calendar event (VRFY-01a)**
1. Open Outlook (web or desktop)
2. Open the resource mailbox calendar (the vehicle's equipment mailbox)
3. Navigate to the booking date
4. Confirm an event exists with:
   - Subject containing the employee name and vehicle info
   - Correct start/end times
   - Body containing booking details (employee, vehicle, location)

**Step 4: Verify personal calendar event (VRFY-02a)**
1. Open your own Outlook calendar
2. Navigate to the booking date
3. Confirm an event exists with:
   - Subject: "Vehicle Rental: [Make] [Model] ([LicensePlate])"
   - Correct start/end times
   - Body containing vehicle details and pickup location

**Report back:**
- Resource calendar event visible? (yes/no + screenshot description)

- Any errors in Azure Functions console? (paste error messages if any)
- Event subjects and times correct?

**If Task 1 reported issues:**
- Analyze the error messages from the Azure Functions console
- Common issues to check:
  - **403 Forbidden**: Missing or unapproved Graph API permissions. Verify Calendars.ReadWrite is granted with admin consent.
  - **404 Not Found on resource mailbox**: The resource mailbox email may be incorrect or the mailbox hasn't finished provisioning. Verify with `Get-Mailbox -Identity <alias>` in Exchange Online PowerShell.
  - **404 Not Found on user**: The LOCAL_DEV_EMAIL may not match a real Entra ID user. Verify the email is correct.
  - **Authentication error**: Check AZURE_TENANT_ID, AZURE_CLIENT_ID, AZURE_CLIENT_SECRET values.
  - **Event created but times wrong**: Check the stripTrailingZ function and UTC timezone handling.
- Fix the root cause in the relevant source file
- Rebuild and restart: `cd api && npm run build && npm start`
- Ask the user to create another booking and re-verify

**If Task 1 passed -- proceed to lifecycle updates:**

Guide the user through verifying cancel, check-out, and check-in updates:

**Cancel verification (VRFY-01b):**
1. Create a new booking for the test vehicle (different time slot)
2. Note the booking ID
3. Cancel it via: `curl -X DELETE http://localhost:7071/api/bookings/<BOOKING_ID>`
4. User checks both calendars: events should now have [CANCELLED] in the subject

**Check-out verification (VRFY-01c):**
1. Create a booking with start time NOW or in the past (check-out window is 30min before to 1hr after start)
2. Check out via: `curl -X PATCH http://localhost:7071/api/bookings/<BOOKING_ID>/checkout`
3. User checks both calendars: events should now have [IN USE] in the subject

**Check-in verification (VRFY-01d):**
1. Using the checked-out booking from above
2. Check in via: `curl -X PATCH http://localhost:7071/api/bookings/<BOOKING_ID>/return`
3. User checks both calendars: events should now have [RETURNED] in the subject

After each step, update the verification checklist Section 4 with pass/fail results.
