  <what-built>
The notification service (notificationService.ts) is already implemented. When a booking is created, `sendBookingNotifications()` dispatches three notifications in parallel:
1. **Email** via Graph `/users/{senderEmail}/sendMail` -- HTML email with booking details
2. **Teams activity feed** via Graph `/users/{userId}/teamwork/sendActivityNotification` -- appears in Teams activity tab
3. **Manager alert** via Graph -- looks up manager via `/users/{id}/manager`, then sends Teams activity notification to the manager

All three are fire-and-forget (errors logged, not thrown).

**If Task 1 reported issues, debug and fix:**

**Email issues:**
- **403 Forbidden on sendMail**: Check that Mail.Send application permission is granted with admin consent. Also verify NOTIFICATION_SENDER_EMAIL is a valid licensed mailbox (not a shared mailbox without a license).
- **404 on sender**: The NOTIFICATION_SENDER_EMAIL user doesn't exist. Verify the email address.
- **Email not arriving**: Check spam/junk folder. Check Azure Functions console for "Booking confirmation email sent" log message. If no log message, the function may not be calling sendBookingConfirmationEmail.
- **Email content wrong**: Fix in `api/src/templates/emailConfirmation.ts`.

**Teams activity feed issues:**
- **403 Forbidden on sendActivityNotification**: Check that TeamsActivity.Send (or TeamsActivity.Send.User) permission is granted. NOTE: The exact permission name varies -- try both `TeamsActivity.Send` and `TeamsActivity.Send.User` in the Azure Portal. Also, the Teams app manifest may need to define activity types. Check if there's a Teams app manifest (manifest.json) that needs `webApplicationInfo` and `activities` sections.
- **404 on user**: The LOCAL_DEV userId may not match a real Entra ID user. The userId comes from the x-ms-client-principal header in production; in local dev it uses LOCAL_DEV_EMAIL. Check if the notification service is using userEmail vs userId correctly.
- **No Teams app installed**: Teams activity feed notifications require the app to be installed in Teams for the recipient user. If using SPFx in Teams, ensure the app is added as a tab. This may be a limitation for local dev testing.
- **activityType not recognized**: The Teams app manifest must declare the activity types (bookingConfirmed, managerBookingAlert, pickupReminder, returnReminder, bookingOverdue). If no manifest declares these, the API call will fail. This may require creating or updating the Teams app manifest.

**Manager notification issues:**
- **No manager found**: Verify the test user has a manager set in Entra ID (self-manager). Check Azure Functions console for "Failed to get manager" error messages.
- **Manager notification sent but not visible**: Both employee and manager notifications go to the same Teams activity feed (since self-manager). They should appear as separate notifications with different activity types.
