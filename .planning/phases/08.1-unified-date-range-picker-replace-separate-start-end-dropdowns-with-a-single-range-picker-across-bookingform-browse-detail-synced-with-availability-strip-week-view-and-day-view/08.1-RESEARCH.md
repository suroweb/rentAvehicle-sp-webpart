## Summary

This phase replaces the current 4-dropdown date/time selection (2 DatePickers + 2 hour Dropdowns) across BookingForm (used in both Browse and Vehicle Detail pages) with a unified range picker: an inline calendar grid with visual range highlighting plus two hour dropdowns below it. The range state must be the single source of truth, bidirectionally synced with the availability strip (week view) and day view. Users can edit the range from any surface -- calendar, strip, or day view -- and all others update immediately.

The critical technical challenge is that **Fluent UI v8 Calendar does NOT support arbitrary date range selection**. Its `dateRangeType` enum only offers Day, Week, Month, and WorkWeek. There is no built-in "click start, click end, highlight between" mode. The recommended approach is to build a custom range calendar component that uses Fluent UI's `Calendar` component with `customDayCellRef` for per-cell styling (start/end circles, range band), combined with custom click-to-select logic managed via React state. This avoids hand-rolling a calendar grid from scratch while getting the visual range highlighting.

The second challenge is adding range overlay and drag handles to the existing AvailabilityStrip and AvailabilityTimeline (day view) components. These currently only display booked/free status and handle single-slot clicks. They need to be extended with: (1) a semi-transparent overlay band showing the selected range, (2) draggable edge handles for fine-tuning, and (3) snap-to-boundary logic when a handle hits a booked slot.

**Primary recommendation:** Build a custom `RangeCalendar` component wrapping Fluent UI v8 `Calendar` with `customDayCellRef` for range styling. Lift range state (`{ startDate, startHour, endDate, endHour }`) to VehicleDetail as the single source of truth. Wire bidirectional callbacks between RangeCalendar, AvailabilityStrip, and AvailabilityTimeline. For drag handles, use `onPointerDown/onPointerMove/onPointerUp` with pointer capture (not mouse events) for cross-device support.

### Pattern 1: Lifted Range State as Single Source of Truth
**What:** A single `IRangeState` interface held in VehicleDetail (or VehicleBrowse for the browse page) that all child components read and write via callbacks.
**When to use:** When multiple sibling components must reflect and edit the same data.
**Example:**
```typescript
// Range state interface -- the single source of truth
interface IRangeState {
  startDate: Date;
  startHour: number;
  endDate: Date;
  endHour: number;
}

// In VehicleDetail:
const [range, setRange] = React.useState<IRangeState>({
  startDate: initialStartDate || getToday(),
  startHour: initialStartHour ?? getNextFullHour(),
  endDate: initialEndDate || getToday(),
  endHour: initialEndHour ?? (getNextFullHour() >= 23 ? 23 : getNextFullHour() + 1),
});

// Partial update helper -- preserves other fields
const updateRange = React.useCallback(function onUpdateRange(partial: Partial<IRangeState>): void {
  setRange(function mergeRange(prev: IRangeState): IRangeState {
    return { ...prev, ...partial };
  });
}, []);

// Pass to all children:
// <RangeCalendar range={range} onRangeChange={updateRange} />
// <AvailabilityStrip range={range} onRangeChange={updateRange} ... />
// <HourDropdowns range={range} onRangeChange={updateRange} ... />
```

### Pattern 2: Custom Range Calendar wrapping Fluent UI Calendar
**What:** A RangeCalendar component that renders Fluent UI's `Calendar` with `customDayCellRef` to apply range styling, and intercepts `onSelectDate` to implement two-click range selection.
**When to use:** When you need range selection but the built-in Calendar only supports single-date or fixed-range types.
**Example:**
```typescript
// Source: Verified from Fluent UI v8 Calendar.types.d.ts -- customDayCellRef on CalendarDayGrid
import { Calendar, DayOfWeek } from '@fluentui/react/lib/Calendar';

interface IRangeCalendarProps {
  range: IRangeState;
  onRangeChange: (partial: Partial<IRangeState>) => void;
  minDate?: Date;
}

const RangeCalendar: React.FC<IRangeCalendarProps> = ({ range, onRangeChange, minDate }) => {
  // Track whether next click sets start or end
  const [selectingEnd, setSelectingEnd] = React.useState<boolean>(false);

  const handleSelectDate = React.useCallback(function onSelect(date: Date): void {
    if (!selectingEnd) {
      // First click: set start date, reset end to same date
      onRangeChange({ startDate: date, endDate: date });
      setSelectingEnd(true);
    } else {
      // Second click: set end date (swap if before start)
      if (date < range.startDate) {
        onRangeChange({ startDate: date, endDate: range.startDate });
      } else {
        onRangeChange({ endDate: date });
      }
      setSelectingEnd(false);
    }
  }, [selectingEnd, range.startDate, onRangeChange]);

  // customDayCellRef: apply CSS classes for start, end, and in-range dates
  const customDayCellRef = React.useCallback(function styleDayCell(
    element: HTMLElement, date: Date
  ): void {
    const dayMs = new Date(date.getFullYear(), date.getMonth(), date.getDate()).getTime();
    const startMs = new Date(range.startDate.getFullYear(), range.startDate.getMonth(), range.startDate.getDate()).getTime();
    const endMs = new Date(range.endDate.getFullYear(), range.endDate.getMonth(), range.endDate.getDate()).getTime();

    // Clear previous range classes
    element.classList.remove('rangeStart', 'rangeEnd', 'inRange');

    if (dayMs === startMs) {
      element.classList.add('rangeStart');
    }
    if (dayMs === endMs) {
      element.classList.add('rangeEnd');
    }
    if (dayMs > startMs && dayMs < endMs) {
      element.classList.add('inRange');
    }
  }, [range.startDate, range.endDate]);

  return (
    <Calendar
      value={range.startDate}
      onSelectDate={handleSelectDate}
      firstDayOfWeek={DayOfWeek.Monday}
      isMonthPickerVisible={false}
      showGoToToday={false}
      minDate={minDate || getToday()}
      calendarDayProps={{ customDayCellRef: customDayCellRef }}
    />
  );
};
```

**What:** A positioned overlay layer on top of the strip grid that spans from the start slot to the end slot, with draggable handles on the edges.
**When to use:** When you need a visual range indicator on a grid that supports user adjustment.

  setDraggingEdge(edge);

const handlePointerMove = React.useCallback(function onPointerMove(
  e: React.PointerEvent
): void {
  if (!draggingEdge) return;
  // Convert pointer position to grid slot coordinates
  // Snap to nearest hour boundary
  // Check for booked slot collision -- snap to boundary
  const slot = positionToSlot(e.clientX, e.clientY);
  if (slot && !isSlotBooked(slot)) {
    if (draggingEdge === 'start') {
      onRangeChange({ startDate: slot.date, startHour: slot.hour });
    } else {
      onRangeChange({ endDate: slot.date, endHour: slot.hour });
    }
  }
}, [draggingEdge, onRangeChange]);

  setDraggingEdge(null);
}, []);

### Anti-Patterns to Avoid
- **Duplicating range state in children:** Each component (calendar, strip, day view) must NOT maintain its own copy of the range. Only the parent holds state. Children receive it as props and call back to update.
- **Using `onSelectDate` with `DateRangeType.Day` and expecting multi-day selection:** Fluent UI Calendar's DateRangeType.Day selects exactly one day. Custom range logic must be built on top.
- **Mouse events for drag handles:** Using `onMouseDown/onMouseMove/onMouseUp` requires separate touch event handling. Use Pointer Events API for unified behavior across devices.
- **Re-rendering the entire strip on every drag pixel:** Throttle/debounce pointer move events or use a ref-based approach for handle position during active drag, then commit to state on pointer up.
- **Clearing range on week navigation:** Per locked decision, the range persists across weeks. The strip overlay should show the visible portion of the range or nothing if the range is outside the visible week.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Calendar month grid | Custom div table with day cells | Fluent UI `Calendar` with `customDayCellRef` | Calendar handles month calculations, leap years, keyboard navigation, accessibility, first-day-of-week, prev/next month navigation. Rewriting this is weeks of work. |
| Date arithmetic | Manual day/month/year math | `@fluentui/date-time-utilities` (`compareDates`, `addDays`, `getMonthStart`) | Date edge cases (month boundaries, DST) are notoriously tricky. The utilities are already bundled. |
| Timezone conversion | New conversion logic | Existing `localToUtcIso` and `useTimezone` hook | Already proven working in the codebase. Reuse exactly. |
| Pointer capture for drag | Manual global event listeners | `element.setPointerCapture(e.pointerId)` | Built-in browser API. Handles all the edge cases of events continuing outside the element during drag. |

**Key insight:** The biggest trap would be building a calendar grid from scratch. Fluent UI v8's Calendar is battle-tested, accessible, keyboard-navigable, and has the `customDayCellRef` hook that makes range styling possible. Build ON it, not around it.

## Common Pitfalls

### Pitfall 1: Fluent UI Calendar customDayCellRef Timing
**What goes wrong:** `customDayCellRef` fires during render. If your range state update triggers a re-render but the Calendar doesn't re-run `customDayCellRef` for already-mounted cells, stale CSS classes remain.
**Why it happens:** React reconciliation may not re-render unchanged Calendar cells, so `customDayCellRef` only fires for newly rendered cells.
**How to avoid:** When range state changes, force the Calendar to re-render by changing the `key` prop or by using `navigatedDate` state. Alternatively, use a `useEffect` that queries all day cells and applies/removes classes imperatively after each range change.
**Warning signs:** Range band "sticks" on old dates after selecting a new range.

### Pitfall 2: Two-Click Selection State Machine
**What goes wrong:** User clicks start date, then navigates to a different month and clicks end date. The "selectingEnd" flag persists incorrectly, or the user gets confused about which click they're on.
**Why it happens:** No visual indicator of selection mode; user forgets they're mid-selection.
**How to avoid:** (1) Visual indicator on the calendar showing "Click end date" or a pulsing start date marker. (2) Clicking the same date as start resets to single-day selection. (3) Allow click anywhere to always be valid -- if new click is before current start, it becomes the new start.
**Warning signs:** Users reporting unexpected range selections.

### Pitfall 3: Drag Handle on Strip -- Coordinate Mapping
**What goes wrong:** Pointer position in pixels doesn't correctly map to hour/day slot because of varying column widths, scroll offset, or zoom.
**Why it happens:** The strip uses flexbox with `flex: 1 1 0` for day columns, so column widths are dynamic.
**How to avoid:** Use `element.getBoundingClientRect()` on the specific day column and hour block to get exact pixel boundaries. Calculate slot from relative position within the day column. Account for `scrollLeft` on the strip container.
**Warning signs:** Drag handle jumps to wrong slot, especially at different window widths.

### Pitfall 4: Range Preserved Across Week Navigation
**What goes wrong:** Navigating to a different week clears the range overlay, or the overlay spans the entire week when it should only show a partial range.
**Why it happens:** The strip re-renders with new `weekOffset` and the overlay calculation doesn't account for range start/end being outside the visible week.
**How to avoid:** Overlay computation must clamp to the visible week boundaries. If `range.startDate` is before the first visible day, clamp overlay start to day 0, hour 0. If `range.endDate` is after the last visible day, clamp overlay end to day 6, hour 23. If both are outside, show no overlay.
**Warning signs:** Overlay appearing on wrong days or spanning the full week when range is only partially visible.

### Pitfall 5: React 17 and Pointer Events
**What goes wrong:** Pointer events not working in SPFx context or in Teams iframe.
**Why it happens:** Some older browsers or sandboxed environments may not support pointer events. React 17 supports pointer events but SPFx 1.22 targets ES5.
**How to avoid:** Pointer Events API is widely supported (all modern browsers). The ES5 target only affects TypeScript compilation, not browser API availability. Test in both SharePoint and Teams tab context. Provide mouse event fallback for Edge Legacy if needed (unlikely in 2026).
**Warning signs:** Drag handles non-functional in specific browsers.

### Pitfall 6: Browse Page vs Detail Page BookingForm
**What goes wrong:** Building two separate range picker implementations -- one for VehicleBrowse filter bar and one for BookingForm in VehicleDetail.
**Why it happens:** VehicleBrowse currently has separate inline DatePickers in its filter bar, and BookingForm has its own separate ones.
**How to avoid:** Extract the range calendar + hour dropdowns into a reusable `RangeCalendar` component. Both VehicleBrowse and BookingForm import and render the same component. The only difference is surrounding context (filter bar vs. booking form).
**Warning signs:** Duplicate range selection logic with subtle behavioral differences between pages.

### Pointer Events for Drag Handles
```typescript
// Source: MDN Pointer Events API -- native browser API, no library needed
// Works for both mouse and touch in a single event model

function DragHandle(props: { edge: 'start' | 'end'; onDrag: (hour: number) => void; onDragEnd: () => void }): React.ReactElement {
  const handlePointerDown = function onDown(e: React.PointerEvent<HTMLDivElement>): void {
    (e.currentTarget as HTMLElement).setPointerCapture(e.pointerId);
  };

  const handlePointerMove = function onMove(e: React.PointerEvent<HTMLDivElement>): void {
    if (!(e.currentTarget as HTMLElement).hasPointerCapture(e.pointerId)) return;
    const hour = pixelToHour(e.clientY, containerRef.current);
    props.onDrag(hour);
  };

  const handlePointerUp = function onUp(e: React.PointerEvent<HTMLDivElement>): void {
    (e.currentTarget as HTMLElement).releasePointerCapture(e.pointerId);
    props.onDragEnd();
  };

  return (
    <div
      className={styles.dragHandle}
      onPointerDown={handlePointerDown}
      onPointerMove={handlePointerMove}
      onPointerUp={handlePointerUp}
    />
  );
}
```

### Snap-to-Boundary Logic
```typescript
// When drag handle reaches a booked slot, find the nearest free boundary

function snapToBoundary(
  targetHour: number,
  bookedHours: Set<number>,
  direction: 'start' | 'end',
  minHour: number,
  maxHour: number
): number {
  if (!bookedHours.has(targetHour)) return targetHour;

  // For start handle: find the first free hour after the booked block
  // For end handle: find the last free hour before the booked block
  if (direction === 'end') {
    // Walk backward to find last free hour
    for (let h = targetHour - 1; h >= minHour; h--) {
      if (!bookedHours.has(h)) return h;
    }
    return minHour;
  } else {
    // Walk forward to find first free hour
    for (let h = targetHour + 1; h <= maxHour; h++) {
      if (!bookedHours.has(h)) return h;
    }
    return maxHour;
  }
}
```

## Open Questions

1. **Calendar customDayCellRef re-render behavior**
   - What we know: The `customDayCellRef` callback fires when cells mount/update. Fluent UI Calendar may not re-call it when only external props change.
   - What's unclear: Whether changing `value` or re-rendering the parent triggers `customDayCellRef` for existing cells or only newly mounted ones.
   - Recommendation: During implementation, test this behavior. If stale classes remain, use a `useEffect` + DOM query approach to update all day cells when range changes. Could also use a `key={range.startDate.getTime() + '-' + range.endDate.getTime()}` to force full re-render.

2. **Strip drag handle precision on mobile**
   - What we know: Strip blocks are 44px tall on mobile (per Phase 8 decision). Drag handles need to be at least 44px touch targets.
   - What's unclear: Whether 44px drag handles will be visually clean when overlaid on the strip's small blocks.
   - Recommendation: Claude's discretion per CONTEXT.md. Suggest using wider handle bars (12px wide, full row height) with semi-transparent grab indicators. Test with 44px minimum.

3. **VehicleBrowse range calendar layout**
   - What we know: VehicleBrowse currently has a horizontal filter bar with DatePickers, Dropdowns, and a Search button.
   - What's unclear: How to fit an inline calendar into the browse page filter bar without it being too tall or taking too much space.
   - Recommendation: On the Browse page, keep the calendar collapsible or in a compact inline layout above the search button. The calendar is more important on the Detail page where it's always visible in the right column. For Browse, a more compact variation may be needed (possibly with the calendar in a smaller section above the hour dropdowns, but still inline/always visible per locked decision).
