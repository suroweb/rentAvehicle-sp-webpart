Create `api/src/models/Report.ts` with TypeScript interfaces:

- `IUtilizationData`: locationId, locationName, vehicleCount, totalBookingHours, totalAvailableHours, utilizationRate (number 0-100)
- `IUtilizationVehicleData`: vehicleId, make, model, licensePlate, totalBookingHours, totalAvailableHours, utilizationRate
- `ITrendData`: period (string YYYY-MM-DD or YYYY-MM-DD week start), bookingCount, utilizationPct
- `IRawBookingRecord`: bookingId, vehicleMake, vehicleModel, vehicleLicensePlate, locationName, startTime, endTime, status, durationHours. NO userId, userDisplayName, userEmail -- privacy first (locked decision).
- `IKpiSummary`: utilizationRate (overall %), totalBookings, activeBookings, overdueCount
- `ITeamBooking`: bookingId, vehicleMake, vehicleModel, vehicleLicensePlate, locationName, locationTimezone, userDisplayName, startTime, endTime, status (manager CAN see names per locked decision)

Create `api/src/services/reportingService.ts` with these exported functions:

1. `getKpiSummary(dateFrom: Date, dateTo: Date)` -> IKpiSummary
   - COUNT bookings in date range with status IN ('Confirmed', 'Active', 'Completed', 'Overdue')
   - COUNT active bookings (status = 'Active')
   - COUNT overdue bookings (status = 'Overdue')
   - Calculate overall utilization: total booking hours / (vehicle count * total hours in range) * 100
   - Handle division by zero (return 0)

2. `getUtilizationByLocation(dateFrom: Date, dateTo: Date)` -> IUtilizationData[]
   - SQL GROUP BY l.id, l.name joining Vehicles and Bookings
   - Clamp booking hours to date range bounds with CASE WHEN (per research pitfall #2)
   - Use DATEDIFF(HOUR, ...) for hour calculation
   - Only count non-archived vehicles with status 'Available'
   - Only count bookings with status IN ('Confirmed', 'Active', 'Completed', 'Overdue')
   - Calculate utilizationRate in TypeScript (avoid SQL division precision issues)

3. `getUtilizationByVehicle(dateFrom: Date, dateTo: Date, locationId: number)` -> IUtilizationVehicleData[]
   - Same pattern as by-location but grouped by vehicle within a specific location
   - Returns make, model, licensePlate per vehicle

4. `getBookingTrends(dateFrom: Date, dateTo: Date, granularity: 'daily' | 'weekly', locationId?: number, categoryId?: number)` -> ITrendData[]
   - GROUP BY date period using CONVERT(VARCHAR(10), b.startTime, 23) for daily
   - For weekly: group by week start using DATEADD(DAY, -(DATEPART(WEEKDAY, b.startTime) - 1), b.startTime)
   - Dynamic WHERE clause for optional locationId and categoryId filters (follow getAllBookings pattern from Phase 4)
   - Return bookingCount per period
   - Calculate utilizationPct per period using total vehicle count and period hours

5. `getRawBookingData(dateFrom: Date, dateTo: Date)` -> IRawBookingRecord[]
   - Select booking details with vehicle and location info
   - MUST NOT include userId, userDisplayName, or userEmail (privacy locked decision)
   - Include computed durationHours via DATEDIFF
   - Order by startTime DESC

6. `getDirectReportIds(managerId: string)` -> string[]
   - Use getGraphClient() from existing graphService.ts
   - Call `/users/${managerId}/directReports` with select('id,userPrincipalName')
   - Return array of Entra ID object IDs
   - Handle empty response gracefully (return [])

7. `getTeamBookings(directReportIds: string[])` -> ITeamBooking[]
   - Return empty array if directReportIds is empty
   - Build parameterized IN clause (follow research example pattern)
   - Query bookings WHERE userId IN (...) AND status IN ('Confirmed', 'Active', 'Overdue') AND endTime > GETUTCDATE()
   - JOIN Vehicles and Locations for display data
   - Include userDisplayName (managers CAN see names per locked decision)
   - ORDER BY startTime ASC

All functions use `getPool()` from existing `./database.js` and `sql` from `mssql`. Follow existing import patterns from bookingService.ts.

Create `api/src/functions/reporting.ts` with 4 admin endpoints following the adminBookings.ts pattern:

1. `GET /api/backoffice/reports/utilization`
   - Auth: requireRole('Admin', 'SuperAdmin')
   - Query params: dateFrom (required), dateTo (required), locationId (optional -- if provided, return per-vehicle data for that location)
   - Parse dateFrom/dateTo as Date objects, validate they are valid dates
   - If locationId is provided: call getUtilizationByVehicle(dateFrom, dateTo, locationId) and return vehicle-level data
   - If no locationId: call getUtilizationByLocation(dateFrom, dateTo) and return location-level data
   - Return 200 with JSON array

2. `GET /api/backoffice/reports/trends`
   - Auth: requireRole('Admin', 'SuperAdmin')
   - Query params: dateFrom (required), dateTo (required), granularity ('daily'|'weekly', required), locationId (optional), categoryId (optional)
   - Call getBookingTrends with all params
   - Return 200 with JSON array

3. `GET /api/backoffice/reports/kpi`
   - Auth: requireRole('Admin', 'SuperAdmin')
   - Query params: dateFrom (required), dateTo (required)
   - Call getKpiSummary(dateFrom, dateTo)
   - Return 200 with JSON object

4. `GET /api/backoffice/reports/export`
   - Auth: requireRole('Admin', 'SuperAdmin')
   - Query params: dateFrom (required), dateTo (required), type ('summary'|'raw', required)
   - If type='summary': call getUtilizationByLocation and return JSON array (frontend handles CSV generation)
   - If type='raw': call getRawBookingData and return JSON array (frontend handles CSV generation)
   - Return 200 with JSON array

All endpoints: try/catch with 500 error response, context.error logging. Use getUserFromRequest + requireRole pattern from adminBookings.ts.
