<objective>
Create the admin provisioning workflow for Exchange equipment mailboxes, the backfill migration endpoint, and a human verification checkpoint for end-to-end calendar integration.

Purpose: Phase 5 requires each vehicle to have an Exchange equipment mailbox (M365-01). The locked decision specifies "auto-provision Exchange equipment mailbox when admin adds a new vehicle." However, Graph API cannot create Exchange mailboxes (see RESEARCH.md, Open Question #2) -- the only path is Exchange Online PowerShell `New-Mailbox -Equipment`. This is a platform limitation, not a design choice. This plan implements the closest achievable workflow: a ready-to-run PowerShell script + admin API endpoints for setting the mailbox email + provisioning status dashboard, making the provisioning flow as streamlined as possible. The checkpoint task asks the user to confirm this two-step approach is acceptable, or to request Azure Automation Runbook integration for true auto-provisioning in a future iteration.

Output: PowerShell provisioning script, admin calendar endpoints (provisioning status + backfill), vehicle service updates for resourceMailboxEmail, and a verification checkpoint confirming the full calendar integration works end-to-end.
</objective>

**1. Create PowerShell provisioning script (scripts/provision-vehicle-mailbox.ps1):**

Create a well-documented PowerShell script that an Exchange admin runs to provision an equipment mailbox for a vehicle. The script should:

- Accept parameters: `-Make`, `-Model`, `-LicensePlate`, `-LocationName`, `-AppServiceAccount` (the app registration's service principal email or UPN, e.g., "rentavehicle-app@contoso.com"), `-RoomListName` (optional, for location grouping)
- Generate a mailbox alias from the vehicle info: `car-{make}-{model}-{licensePlate}` (lowercase, alphanumeric only, hyphens allowed)
- Generate a display name: `{Make} {Model} ({LicensePlate}) - {LocationName}`
- Run `New-Mailbox -Equipment -Name "{displayName}" -DisplayName "{displayName}" -Alias "{alias}"`
- Configure calendar processing:
  ```powershell
  Set-CalendarProcessing -Identity "{alias}" `
    -AutomateProcessing AutoAccept `
    -AllBookInPolicy $false `
    -BookInPolicy "{AppServiceAccount}" `
    -AllowConflicts $false
  ```
- Grant all org users Reviewer access to the calendar:
  ```powershell
  Set-MailboxFolderPermission -Identity "{alias}:\Calendar" -User Default -AccessRights Reviewer
  ```
- Optionally create/add to a room list distribution group for location grouping:
  ```powershell
  if ($RoomListName) {
    # Try to create the room list (ignore error if already exists)
    try { New-DistributionGroup -Name $RoomListName -RoomList -ErrorAction Stop }
    catch { Write-Host "Room list '$RoomListName' already exists or cannot be created." }
    Add-DistributionGroupMember -Identity $RoomListName -Member "$alias@$domain"
  }
  ```
- Output the generated email address for the admin to enter in the app.
- Include a header comment block explaining: prerequisites (ExchangeOnlineManagement module, Connect-ExchangeOnline), usage examples, and what the script does.

**3. Update vehicleService.ts (api/src/services/vehicleService.ts):**

- Add a new function `updateVehicleMailbox(vehicleId: number, resourceMailboxEmail: string): Promise<boolean>`:
  ```typescript
  export async function updateVehicleMailbox(vehicleId: number, resourceMailboxEmail: string): Promise<boolean> {
    const pool = await getPool();
    const result = await pool.request()
      .input('vehicleId', sql.Int, vehicleId)
      .input('resourceMailboxEmail', sql.NVarChar(255), resourceMailboxEmail)
      .query('UPDATE Vehicles SET resourceMailboxEmail = @resourceMailboxEmail, updatedAt = GETUTCDATE() WHERE id = @vehicleId AND isArchived = 0');
    return result.rowsAffected[0] > 0;
  }
  ```

**4. Update vehicles.ts endpoints (api/src/functions/vehicles.ts):**

Add a new admin endpoint for setting a vehicle's resource mailbox email:

```typescript
// PATCH /api/backoffice/vehicles/{id}/mailbox
// Sets the resource mailbox email for a vehicle after Exchange provisioning
```

This endpoint:
- Requires Admin or SuperAdmin role
- Accepts body `{ resourceMailboxEmail: string }` with basic email validation
- Calls `updateVehicleMailbox(id, resourceMailboxEmail)`
- Returns 200 on success, 404 if vehicle not found

Also ensure the existing vehicle create/update endpoints pass `resourceMailboxEmail` through if provided in the input body.

**5. Create calendarAdmin.ts (api/src/functions/calendarAdmin.ts):**

Create a new function module with 2 admin endpoints:

**GET /api/backoffice/calendar/status** -- Provisioning status:
- Requires Admin or SuperAdmin role
- Query: `SELECT v.id, v.make, v.model, v.licensePlate, v.resourceMailboxEmail, l.name as locationName FROM Vehicles v JOIN Locations l ON v.locationId = l.id WHERE v.isArchived = 0 ORDER BY l.name, v.make, v.model`
- Returns array of vehicles with their provisioning status (has resourceMailboxEmail or not)
- Includes summary: `{ total: N, provisioned: M, unprovisioned: N-M, vehicles: [...] }`

**POST /api/backoffice/calendar/backfill** -- Backfill existing bookings to calendars:
- Requires SuperAdmin role (destructive/bulk operation)
- Query bookings that have no vehicleCalendarEventId AND no employeeCalendarEventId AND status IN ('Confirmed', 'Active', 'Overdue') -- these are live bookings without calendar events
- Process in batches of 10 with 2-second delays between batches to respect Graph API rate limits
- For each booking, call `syncBookingToCalendars(bookingId, 'created')` from calendarService
- Return a summary: `{ total: N, synced: M, failed: F, errors: [...] }`
- Log progress: `context.log('Backfill progress: {i}/{total}')`
