**UserContext model (`api/src/models/UserContext.ts`):**

Create the `UserContext` interface:
```typescript
export interface UserContext {
  userId: string;       // oid claim from token
  displayName: string;  // name claim
  email: string;        // name_typ claim value
  roles: string[];      // all role claim values from token
  effectiveRole: 'Admin' | 'Manager' | 'Employee';
}
```

Also export `AppRole` type: `'Admin' | 'Manager' | 'Employee'`

**Auth middleware (`api/src/middleware/auth.ts`):**

Implement per the research doc Pattern 3:

1. `parseClientPrincipal(headerValue: string): UserContext | null`
   - Base64-decode the `x-ms-client-principal` header
   - Parse the JSON structure (auth_typ, claims array, name_typ, role_typ)
   - Extract userId from `oid` or `nameidentifier` claim
   - Extract displayName from `name` claim
   - Extract email from the claim matching `name_typ`
   - Extract all role claims matching `role_typ`
   - Apply hierarchy: Admin > Manager > Employee for `effectiveRole`
   - If no roles in token, default to `Employee`
   - Return null on any parse error (malformed header)

2. `requireRole(...allowedRoles: AppRole[]): (user: UserContext | null) => boolean`
   - Returns false if user is null
   - Returns true if user's effectiveRole is in allowedRoles

3. `getUserFromRequest(request: HttpRequest): UserContext | null`
   - Reads `x-ms-client-principal` header from request
   - Calls parseClientPrincipal
   - Returns null if header missing or invalid

4. **Local development mock:** Add a function `getLocalDevUser(): UserContext | null`
   - Only active when `process.env.LOCAL_DEV === 'true'`
   - Returns a mock user from env vars: `LOCAL_DEV_NAME`, `LOCAL_DEV_EMAIL`, `LOCAL_DEV_ROLE` (defaults to Employee)
   - Update `getUserFromRequest` to fall back to `getLocalDevUser()` when header is missing AND `LOCAL_DEV=true`
   - Add to `api/local.settings.json`: `"LOCAL_DEV": "true"`, `"LOCAL_DEV_NAME": "Test User"`, `"LOCAL_DEV_EMAIL": "test@contoso.com"`, `"LOCAL_DEV_ROLE": "Admin"`

**Health endpoint (`api/src/functions/health.ts`):**

Two routes:
1. `GET /api/health` -- Public health check. Returns `{ status: "healthy", timestamp: ISO string }`. No auth required. authLevel: "anonymous".
2. `GET /api/admin/health` -- Admin-only health check. Extracts user via `getUserFromRequest`. If no user, return 401. If `requireRole('Admin')` fails, return 403 with `{ error: "Forbidden: Admin role required" }`. On success, return `{ status: "healthy", role: effectiveRole, timestamp: ISO string }`.

**Me endpoint (`api/src/functions/me.ts`):**

1. `GET /api/me` -- Authenticated endpoint. Extracts user via `getUserFromRequest`. If no user, return 401 with `{ error: "Not authenticated" }`. On success, return `{ userId, displayName, email, role: effectiveRole }`.

All endpoints use `app.http()` with `authLevel: 'anonymous'` (Easy Auth handles actual auth at the platform level; function code handles role enforcement).

Use `zod` to validate the client principal structure in `parseClientPrincipal` for extra safety.
