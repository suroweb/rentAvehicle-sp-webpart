**ApiService extensions (services/ApiService.ts):**
Add ALL new methods for Phase 4 (both employee and admin) following existing patterns. Use `this.apiClient.fetch()` for PATCH/DELETE per established pattern (AadHttpClient.fetch() for non-GET/POST):

1. `checkOutBooking(bookingId: number): Promise<void>` -- PATCH `/api/bookings/${bookingId}/checkout`
2. `checkInBooking(bookingId: number): Promise<void>` -- PATCH `/api/bookings/${bookingId}/return`
3. `getTimeline(locationId: number, date: string): Promise<ITimelineData>` -- GET `/api/vehicles/timeline?locationId=${locationId}&date=${date}`
4. `getAllBookings(filters: { locationId?: number; status?: string; dateFrom?: string; dateTo?: string; employeeSearch?: string }): Promise<IBooking[]>` -- GET `/api/backoffice/bookings` with query params built from filters
5. `adminCancelBooking(bookingId: number, cancelReason: string): Promise<void>` -- DELETE `/api/backoffice/bookings/${bookingId}` with JSON body `{ cancelReason }`

Also update `postWithConflict` helper: currently it throws `Error('CONFLICT: ...')` on 409. Change it to parse the full JSON response body on 409 and return a structured object `{ conflict: true, message: string, suggestions: IBookingSuggestion[] }` instead of throwing. This requires updating the return type and callers. Alternatively, if changing the return type is too disruptive, enhance the error to include suggestions data -- attach it as a property on the Error object or use a custom error class. The simplest approach: make postWithConflict return `Promise<T | { conflict: true; message: string; suggestions: IBookingSuggestion[] }>` and let BookingForm check the response.

**IMPORTANT ES5 constraints:** All new code must follow established Phase 3 ES5 patterns: use `function` keyword instead of arrow functions in callbacks, string concatenation instead of template literals, indexed for loops instead of for...of, manual pad2() instead of padStart.
